# wechat2all Codex GUI Bridge

GUI-first bridge from the wechat2all `codex` route to Codex desktop threads.
The default path injects through the visible GUI; Codex app-server discovers
threads, observes results, and provides fallback delivery.

## What This Package Owns

- Connecting to Codex app-server JSON-RPC.
- Listing bindable Codex chats/threads.
- Reading a thread and current binding.
- Binding a WeChat route conversation to a specific Codex thread id.
- Reading token usage from `account/rateLimits/read`.
- Sending prompts to the bound thread.
- Filtering Codex turn output with reply modes: `final`, `silent`, and `stream`.
- Persisting the Codex GUI auto-open preference used by `/autoopen 1|0`.
- Opening the Codex desktop app at project boot when auto-open is enabled.
- Persisting a `/alarm <HH:mm>` keepalive timer that sends a silent dummy
  `你好` prompt to the bound Codex chat once per day.
- Sending inbound WeChat text, images, and files through the exact bound GUI
  chat, including local path references for attachments.
- Returning local files/images generated by Codex so runtime can send them back
  to WeChat.
- GUI automation that opens the exact bound thread, pastes attachments/text,
  presses Return, and polls that same thread for the final answer.
- Creating `/new` chats through app-server, then opening the materialized thread
  in the desktop GUI after its first message completes.

It does not own WeChat routing, the main assistant, or dashboard endpoints.
Those live in `runtime` and `router-daemon`.

## Delivery Modes

`app-server`:

- Uses `thread/start` to create a new chat in the bound project's `cwd`.
- Uses `thread/resume` and `turn/start` to continue a chat.
- Protocol-first and does not require Accessibility permission.
- May not visually type into the currently open GUI window.
- Supports image input through `localImage`. Other files are referenced by
  absolute local path inside the prompt text.

`gui-automation`:

- Opens the bound Codex GUI thread with `codex://threads/<threadId>`.
- Pastes attachment paths and prompt text, then presses Return.
- Uses app-server only to observe the completed GUI turn, or to deliver when GUI
  injection fails before a turn appears.
- Creates `/new` through app-server and opens it after the first completed turn.
- Requires macOS Accessibility/Automation permission for the process that starts
  wechat2all, because it controls System Events and the ChatGPT/Codex GUI.
- Is the default delivery mode for the wechat2all Codex route.

## Reply Modes

`final` is the default. It only returns `final_answer` text to WeChat and drops
thinking/commentary items.

`silent` waits for the Codex turn to complete, then lets the runtime send only a
short completion notice.

`stream` returns every completed Codex assistant text part for that turn. The
runtime maps these parts to multiple WeChat text actions.

## Tech Stack

- TypeScript.
- Node.js child process and stdio/socket JSON-RPC transport.
- Codex app-server protocol.
- macOS custom-URL opening and AppleScript GUI injection.

```bash
pnpm --filter @wechat2all/codex-gui-bridge build
pnpm --filter @wechat2all/codex-gui-bridge dev -- ls
pnpm --filter @wechat2all/codex-gui-bridge dev -- token

pnpm --filter @wechat2all/codex-gui-bridge dev -- alarm off
```

## Setup Check

The package owns a read-only setup checker for the shared Codex prerequisites,
app-server, and optional GUI automation permissions:

```bash
./packages/codex-gui-bridge/scripts/check.sh
./packages/codex-gui-bridge/scripts/check.sh --probe
```

The passive check does not prompt for macOS permissions. `--probe` connects to
Codex app-server, reads the task list, and sends read-only AppleScript events to
System Events and ChatGPT/Codex. It does not paste text, simulate keys, or send
a Codex prompt, but the first Automation request can display a macOS permission
dialog.

After router-daemon is listening and all routes are initialized, the router
starts the `--probe` check asynchronously if the Codex route is installed and
enabled. It streams output to the startup terminal with a
`[codex-route/setup-check]` prefix. A failed check does not prevent WeConnect or
other routes from starting. Set `WECHAT2ALL_CODEX_SETUP_CHECK=0` to disable this
startup check temporarily.

`autoopen 1` is the same persisted setting exposed in the WeChat `codex` route
as `/autoopen 1`. `autoopen 0` disables it. The default is disabled.
`bind` also persists the selected thread locally, so daemon/desktop restarts do
not require another `/bind`. `WECHAT2ALL_CODEX_THREAD_ID` remains an explicit
environment override.

## Environment

```text
WECHAT2ALL_CODEX_DELIVERY=gui-automation
# WECHAT2ALL_CODEX_DELIVERY=app-server
WECHAT2ALL_CODEX_THREAD_ID=<optional-prebound-thread-id>
WECHAT2ALL_CODEX_REPLY_MODE=final
# WECHAT2ALL_CODEX_REPLY_MODE=silent
# WECHAT2ALL_CODEX_REPLY_MODE=stream
WECHAT2ALL_CODEX_APP_SERVER_SOCKET=<optional-control-socket>
WECHAT2ALL_CODEX_APP_SERVER_TIMEOUT_MS=8000
WECHAT2ALL_CODEX_TURN_TIMEOUT_MS=180000
WECHAT2ALL_CODEX_MAX_TURN_WAIT_MS=1800000
WECHAT2ALL_CODEX_IN_PROGRESS_GRACE_MS=120000
WECHAT2ALL_CODEX_COMPACTION_GRACE_MS=180000
WECHAT2ALL_CODEX_GUI_POLL_INTERVAL_MS=1000
WECHAT2ALL_CODEX_GUI_THREAD_OPEN_DELAY_MS=900
WECHAT2ALL_CODEX_GUI_NEW_CHAT_DISCOVERY_MS=15000
WECHAT2ALL_CODEX_LIST_LIMIT=20
WECHAT2ALL_CODEX_GUI_BINDING_FILE=<optional-binding-state-file>
WECHAT2ALL_CODEX_GUI_AUTOOPEN_FILE=<optional-autoopen-state-file>
WECHAT2ALL_CODEX_GUI_ALARM_FILE=<optional-alarm-state-file>
WECHAT2ALL_CODEX_AUTOOPEN=1
# Optional overrides. By default the bridge discovers the current ChatGPT.app
# first, then the legacy Codex.app.
WECHAT2ALL_CODEX_GUI_APP_NAME=ChatGPT
WECHAT2ALL_CODEX_GUI_APP_PATH=/Applications/ChatGPT.app
WECHAT2ALL_CODEX_GUI_PROCESS_NAME=ChatGPT
WECHAT2ALL_CODEX_GUI_BUNDLE_ID=<optional-bundle-id>
```

`WECHAT2ALL_CODEX_AUTOOPEN=1` is an environment override. Without it, the bridge
reads the persisted `/autoopen` setting from local runtime state.
Binding, auto-open, and alarm state default to
`~/.wechat2all-runtime-bot/codex-gui-bridge/` with private file permissions.

## Collaborator Notes

- app-server remains required for discovery, result observation, fallback
  delivery, token usage, and `/new`.
- `/status` never probes Desktop IPC. It combines the bridge's current active
  request state with the bound App Server thread and its latest turn state.
- A successful `osascript` exit is provisional: if app-server cannot observe a
  new GUI turn within `WECHAT2ALL_CODEX_GUI_TURN_OBSERVATION_MS` (10 seconds by
  default), the complete request is retried through app-server. This keeps
  locked or display-asleep Macs usable remotely.
- Do not write directly into Codex SQLite state.
- Keep `gui-automation` GUI-first: app-server delivery is a fallback, not the
  normal prompt path.
